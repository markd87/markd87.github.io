Machine Learning mistakes

Non-obvious pitfalls to avoid in machine learning

Aug 8, 2020

machine learning

  • Test data leakage - Feature selection
  • Test data leakage - pre-processing
  • Out of sample prediction
  • Inference confidence
  • One-hot encoding for tree based models

Test data leakage - Feature selection

When choosing features for a regression or classification task out of a large number of potential features, one may first find the most correlated features with the target in the data set, and then build a model and train on a train-test split.

The unobvious mistake is that when choosing the correlated features out of the whole dataset, before doing the train-test split, there is already information leakage from the test set to the training process, leading to potentially wrong conclusions.

To demonstrate it, consider completeley random data set of 1000 records, with 5000 features populated with random values from a normal distribution and a random target. We then select the top 100 features most correlated with the target are selected to train a model.

Regression: Here we use the F-statistic as the measure of correlation between the features and target.

Create randomly distributed features and target:

np.random.seed(0)
X = np.random.rand(1000, 5000)
y = np.random.rand(1000)

First method (wrong): selecting best features then perform train-test split, train a linear regression model and evaluate.

X_new = SelectKBest(f_regression, k=100).fit_transform(X, y)

x_train, x_test, y_train, y_test = train_test_split(X_new, y, test_size=0.33, random_state=42)

model = LinearRegression().fit(x_train, y_train)
print(f"Model R^2 score: {model.score(x_test, y_test):.3f}")

Model R^2 score: 0.328

Second method: create a train-test split, then choose best features from training set, train a linear regression model and evalute.

x_train, x_test, y_train, y_test = train_test_split(X, y, test_size=0.33, random_state=42)

X_new = SelectKBest(f_regression, k=100).fit_transform(x_train, y_train)

model = LinearRegression().fit(x_train, y_train)
print(f"Model R^2 score: {model.score(x_test, y_test):.3f}")

Model R^2 score: -0.118

We can see that the first method has a positive score for the linear regression model which is spurious and non-existed given we used comlpetley random data, whereas the second method has a negative $R^2$ indicating no linear correlation. The non-negative correlation obtained in the first method is due to the test data leakage during the feature selection process.

We can repeat the process using cross validation as well:

First method: select features then perform cross validation:

X_new = SelectKBest(f_regression, k=100).fit_transform(X, y)

model = LinearRegression()

print(f"Cross validation mean score: {cross_val_score(model, X_new, y, cv=5).mean():.3f}")

Cross validation mean score: 0.311

Second method: Perform feature selection in each cross validation split:

select_k = SelectKBest(f_regression, k=100)
model = LinearRegression()

pipe = Pipeline(steps=[('k_best', select_k), ('model', model)])

print(f"Cross validation mean score: {cross_val_score(pipe, X, y, cv=5).mean():.3f}")

Cross validation mean score: -0.340

Classification

The problem can also be seen with a classification problem. Here we use ANOVA F-value statstic to measure correlation between feature and target.

X = np.random.randn(1000, 5000)
y = np.random.randint(low=0, high=2, size=(1000,)) # target classes = [0,1]

First method: selecting best featuers on entire dataset.

X_new = SelectKBest(f_classif, k=50).fit_transform(X, y)

model = LogisticRegression(solver='lbfgs')

print(f"Cross validation mean score: {cross_val_score(model, X_new, y, cv=10).mean():.3f}")

Cross validation mean score: 0.713

Second method: Selecting best features within each training fold.

select_k = SelectKBest(k=50)
model = LogisticRegression(solver='lbfgs')

pipe = Pipeline(steps=[('k_best', select_k), ('model', model)])

print(f"Cross validation mean score: {cross_val_score(pipe, X, y, cv=5).mean():.3f}")

Cross validation mean score: 0.479

As expected the model performans is as good as random, expected given the random dataset.

The difference in the scores between the two methods is significant. The test data leakage results in a much better than random model, simply because the model has indirectly been trained on the test set through the feature selection process.

Test data leakage - pre-processing

A common step in a machine learning training pipline is pre-processing of the input data, e.g. feature standarsation, dimensionality reduction, etc. Following the pre-processing a cross validation is applied to train and evaluate the model. A k-fold cross validation seem to suggest we can be confident in the model's evaluation and performance on future unseen data. However, the issue with this pipeline is that the pre-processing is applied on the whole dataset, such that information from the validation fold is leaked into each training fold, leading to a biased estimation of the model's performance.

from sklearn import datasets, linear_model
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline
from sklearn.linear_model import LogisticRegression
from sklearn.decomposition import PCA
from sklearn.model_selection import StratifiedKFold, KFold, cross_validate

# Load breast cancer dataset
breast = datasets.load_breast_cancer()

breast.feature_names

array(['mean radius', 'mean texture', 'mean perimeter', 'mean area',
       'mean smoothness', 'mean compactness', 'mean concavity',
       'mean concave points', 'mean symmetry', 'mean fractal dimension',
       'radius error', 'texture error', 'perimeter error', 'area error',
       'smoothness error', 'compactness error', 'concavity error',
       'concave points error', 'symmetry error',
       'fractal dimension error', 'worst radius', 'worst texture',
       'worst perimeter', 'worst area', 'worst smoothness',
       'worst compactness', 'worst concavity', 'worst concave points',
       'worst symmetry', 'worst fractal dimension'], dtype='<U23')
X = np.array(breast['data'])
y = np.array(breast['target'])

cv = StratifiedKFold(n_splits = 5, shuffle=True, random_state=0)

Wrong way

Here we perform scaling and PCA before cross validation, resulting in test data leakage.

scaler = StandardScaler().fit(X)
X_scaled = scaler.transform(X)
pca = PCA(3)
X_pca = pca.fit_transform(X_scaled)
clf = LogisticRegression()

score = cross_validate(clf, X_pca, y, cv=cv, scoring=['accuracy'])
print(f"Average accuracy: {score['test_accuracy'].mean():.3f}")

Average accuracy: 0.951

Right way

By using the scikit-learn Pipeline object and including the scaling and PCA as steps, when we pass the pipeline into a cross-validation object, we ensure that the pipeline is run separatley for each training fold, preventing test set leakage.

steps = [('scaler', StandardScaler()), 
         ('pca', PCA(3)), 
         ('logistic', LogisticRegression())]

# Create Pipeline 
pipe = Pipeline(steps)

# Run cross validation with accuracy scoring metric
score = cross_validate(pipe, X, y, cv=cv, scoring=['accuracy'])
print(f"Average accuracy: {score['test_accuracy'].mean():.3f}")

Average accuracy: 0.949

Out of sample prediction

This mistake applies to tree based models (random forest, xgboost, light gbm etc.) in which one tries to obtain a prediction outside of the training domain of the model. This is particularly evident in casess where there is a time-dependent feature and a trend in the data

X = np.linspace(0, 10, 100)
y = 2*X + np.random.randn(100)

plt.scatter(X, y);
plt.xlabel('X')
plt.ylabel('y')
plt.title('Dataset');

Train-test split:

X_train = X[:int(len(X)*0.7)]
y_train = y[:int(len(X)*0.7)]

X_test = X[int(len(X)*0.7):]
y_test = y[int(len(X)*0.7):]

plt.scatter(X_test, y_test, label='test', color='red');
plt.xlabel('X')
plt.ylabel('y')
plt.title('Dataset');
plt.legend();

Train a random forest model and evaluate on test set:

rf = RandomForestRegressor(n_estimators=10, max_depth=3).fit(X_train[:, np.newaxis], y_train)

Performance on training data:

We see in the plot that the model is incapable of predicting y for x values outside its training domain, resulting in the flat black prediction line corresponding to the prediction for the x value at the boundary of the trainin set. This is despite the model being able to fit well the training data, shown with the purple line.

Inference confidence

This issue comes up in classification problems, where a classification model is fit on a data set within a limited domain, however inference is made far away from the training domain, with the model's output treated with the same confidence independent on where the inference is main in the feature space.

To ilustrate this, we construct a random data set consisting of 3 classes and 2 features as shown below.

from sklearn.svm import LinearSVC, SVC
from sklearn.mixture import GaussianMixture

from sklearn.datasets import make_blobs
X, y = make_blobs(n_samples=1000, centers=3, n_features=2, random_state=0)

h=0.05
x_min, x_max = X[:, 0].min() - .5, X[:, 0].max() + .5
y_min, y_max = X[:, 1].min() - .5, X[:, 1].max() + .5
xx, yy = np.meshgrid(np.arange(x_min, x_max, h),
                     np.arange(y_min, y_max, h))

colors = {0: '#FF4C4C', 1:'#34BF49', 2:'#0099E5'}

for cl in np.unique(y):
    idx = np.where(y == cl)
    plt.scatter(X[idx,0],X[idx,1], c=colors[cl], label=cl)
plt.legend(frameon=False,bbox_to_anchor=(1.15,1.05), title='class')
plt.tight_layout()

We start with a linear support vector classifier (SVC) applied on a multi-class problem (one-versus-rest)

clf = LinearSVC(multi_class='ovr')
clf.fit(X, y)

Plotting the model's prediction against the data set we get a good separation of the classes

Z = clf.predict(np.c_[xx.ravel(), yy.ravel()])
Z = Z.reshape(xx.shape)

plt.contourf(xx, yy, Z, levels=[-1,0,1,2],  extend='both', colors=('#FF4C4C', '#34BF49', '#0099E5'), alpha=0.85);
for cl in np.unique(y):

plt.legend(frameon=False,bbox_to_anchor=(1.15,1.05), title='class')
plt.tight_layout()

The problem is evident, however, when we zoom out from the training data domain as ilustrated below.

h=0.05
x_min, x_max = -15, 15
y_min, y_max = -15, 20

We see that it doesn't make sense to say that we are as confident on classifying a point far away from where the training data lies as we are on a point in the center of one of the blobs

An approach is using Gaussian Mixture models, where a number of Gaussians parameters are fitted to the data points using the Expectation-Maximization algorithm.

clf = GaussianMixture(n_components=3)

clf.fit(X,y)

h=0.5
x_min, x_max = X[:, 0].min() - .5, X[:, 0].max() + .5
y_min, y_max = X[:, 1].min() - .5, X[:, 1].max() + .5
xx, yy = np.meshgrid(np.arange(x_min, x_max, h),
                     np.arange(y_min, y_max, h))

XX = np.array([xx.ravel(), yy.ravel()]).T

Z = clf.score_samples(XX)
Z = Z.reshape(xx.shape)

The probability of the observed data under the model:

plt.contourf(xx, yy, Z, levels=50, cmap='viridis', alpha=0.9, linestyles='solid')
plt.colorbar(shrink=0.8)
plt.contour(xx, yy, Z, levels=30,colors='k', linestyles='solid', alpha=0.5)

for cl in np.unique(y):
    idx = np.where(y == cl)
    plt.scatter(X[idx,0],X[idx,1], c=colors[cl], label=cl, edgecolor='k', alpha=0.3) 
    
plt.title('Log likelihood for gaussian mixture given data');

A helper function to plot 2-D Gaussians:

def gauss2d(x, means, cov):
    """
    2d Gaussian function with given means vector
    and covarians matrix cov
    """
    x = np.array(x)
    xshift = means - x
    invc = np.linalg.inv(cov)
    det = np.linalg.det(cov)
    return np.diag(np.exp(-0.5*np.dot(xshift.dot(invc),xshift.transpose()))/np.sqrt(2*np.pi*det))

Get probability density function (pdf) for each of the Gaussian components:

Z1= gauss2d(XX,clf.means_[0],clf.covariances_[0])
Z1 = Z1.reshape(xx.shape)

Z2= gauss2d(XX,clf.means_[1],clf.covariances_[1])
Z2 = Z2.reshape(xx.shape)

Z3= gauss2d(XX,clf.means_[2],clf.covariances_[2])
Z3 = Z3.reshape(xx.shape)

Plotting the pdf for the Gaussian mixture:

pdf = plt.contourf(xx,yy,Z1+Z2+Z3, cmap='RdBu', levels=20)
plt.tight_layout()
plt.colorbar(pdf, label='pdf', shrink=0.8)

for cl in np.unique(y):
    idx = np.where(y == cl)
    plt.scatter(X[idx,0],X[idx,1], c=colors[cl], label=cl, edgecolor='k', alpha=0.2)
    
plt.title('PDF for the gaussian mixture');

The Gaussians are centered on the three blobs, with the PDF decaying quickly outside the available data domain, indicating no confidence for classifying data points far from the traininig data.

The prediction process in this case can then use the PDF or log likelihood to provide the confidence in the model's prediction, or set a threshold beyond which predictions are simply not made because they are far from the model's training domain.

One-hot encoding for tree based models

When we have a combination of high cardinality categorical feature and continuous value features, a typical thing to do is to one-hot encode the categorical feature resulting in multiple new features. The issue with this technique as opposed to for example label encoding the categorical feature, is that following one-hot encoding all newly generated features are independent despite all coming from the same original feature.

What this can cause is that the tree-base model will favour (based on Gini criteria for example) to use the continous feature for splits even if the categorical feature may be more benefitial for the classification problem.
